# Performance Notes

## Dataset Size

The assessment requires the system to support an organization with 10,000 employees.

This dataset size is manageable for PostgreSQL, but inefficient implementation choices can still become visible in API responses, salary insights, and seed generation.

## Employee Listing

The employee list endpoint is paginated.

Default page size:

```text
25
```

Maximum page size:

```text
100
```

This prevents the API from returning all 10,000 employees in one response.

## Salary Insights

Salary insights are calculated using database-level aggregation:

```sql
COUNT(*)
MIN(salary)
MAX(salary)
AVG(salary)
```

The query object uses a single aggregate query instead of loading employee records into Ruby memory.

Conceptually, the query is:

```sql
SELECT
  COUNT(*),
  MIN(salary),
  MAX(salary),
  AVG(salary)
FROM employees
WHERE country = 'India'
  AND job_title = 'Software Engineer';
```

This keeps salary insight calculations efficient and database-driven.

## Indexing Strategy

Salary insight queries commonly filter by:

- country
- job title
- country and job title together

Indexes were added for:

- `employees.country`
- `employees.job_title`
- `employees.country + employees.job_title`

This supports the expected query patterns.

## Seed Generation

The seed script creates 10,000 employees by default.

Full names are generated by combining values from:

```text
db/seed_data/first_names.txt
db/seed_data/last_names.txt
```

The seed implementation is performance-conscious:

- uses `insert_all!` instead of `create!` in a loop
- inserts records in batches
- validates generated batches before insert
- wraps the operation in a transaction
- supports configurable employee count and batch size

Default behavior:

```sh
rails db:seed
```

Custom behavior:

```sh
EMPLOYEE_SEED_COUNT=100 EMPLOYEE_SEED_BATCH_SIZE=25 rails db:seed
```

## Why Bulk Insert

A naive implementation would call:

```ruby
Employee.create!(...)
```

10,000 times.

That would run 10,000 individual insert operations and ActiveRecord validations.

This implementation builds batches and uses:

```ruby
Employee.insert_all!(records)
```

This reduces database round trips significantly.

## Transaction Safety

The seed operation is wrapped in a transaction.

If one batch fails, the database rolls back to the previous state instead of being left with partially seeded data.
